# Стратегия тестирования parse_brsccp.py

## 1. Цели тестирования
- Обеспечить точность извлечения физических величин (энергии, ddG, сдвиги).
- Проверить надежность обработки специфических форматов лога (переносы строк, наследование времени).
- Гарантировать корректную работу CLI и обработку ошибок парсинга.

## 2. Инструментарий и структура
- **Фреймворк:** `unittest` (стандартная библиотека Python, согласно ограничениям проекта).
- **Расположение:** `tests/test_parser.py`.
- **Данные:**
    - Использование существующего `brsccp2V8X.log` для интеграционных тестов.
    - Создание фикстур (строковых фрагментов) для модульных тестов.

## 3. Уровни тестирования

### 3.1 Модульные тесты (Unit Tests)
Каждая функция парсинга должна быть протестирована изолированно:
- `test_preprocess_timestamp_inheritance`: Проверка, что строки без меток получают время от предыдущих.
- `test_parse_input_params_multiline`: Проверка склейки параметров, таких как `remove_by_resname`.
- `test_ast_literal_eval_safety`: Проверка парсинга словарей, списков и чисел.
- `test_parse_results_abnormal`: Проверка возврата `None` при отсутствии блока итогов.

### 3.2 Интеграционные тесты (Integration Tests)
- `test_full_log_processing`: Полный цикл от чтения файла до получения объекта `BrsccpLog`.
- `test_json_csv_consistency`: Проверка, что данные в JSON и CSV форматах идентичны по смыслу.

### 3.3 Тестирование CLI
- `test_cli_strict_mode`: Проверка кода выхода (exit code) при наличии `parse_warnings`.
- `test_cli_output_file`: Проверка корректности записи результата в файл через `--out`.

## 4. Граничные случаи (Edge Cases)
- Пустой файл лога.
- Лог с `ABNORMAL TERMINATION`.
- Файл с неожиданными символами или прерванный посередине блока параметров.
- Несколько блоков `Done: X / N` (убедиться, что парсер берет финальные результаты).

## 5. Запуск тестов
Команда для запуска:
```bash
python3 -m unittest tests/test_parser.py
```
